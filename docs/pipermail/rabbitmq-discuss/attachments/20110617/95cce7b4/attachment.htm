Hi Jon,<div><br><div><meta http-equiv="content-type" content="text/html; charset=utf-8">Rabbit will not scale to a large throughput of messages published to a single queue as they all get delivered to a single node. </div>

<div><br></div><div>Here&#39;s one way to do it:</div><div><br></div><div><meta http-equiv="content-type" content="text/html; charset=utf-8"><a href="http://lists.rabbitmq.com/pipermail/rabbitmq-discuss/2011-March/012007.html">http://lists.rabbitmq.com/pipermail/rabbitmq-discuss/2011-March/012007.html</a></div>

<div><br></div><div>The gist is:</div><div><ul><li>Rabbit doesn&#39;t scale on per queue basis. The answer: create many queues, bound to the same exchange</li><li>Use a direct exchange, which will rely on routing keys to decide which queue to deliver a message to</li>

<li>Pre-declare the queues on the nodes inside your cluster, bind them using a known set of routing keys (eg: queueA with routing keys 1-100 on server1, queueB with routing keys 101-200 on server2, etc...). Assuming durable queues here...</li>

<li>Have your publisher randomly pick a routing key and node to connect to (or use a load balancer)</li><li>Use the mandatory flag to ensure your message reaches a queue, if it fails re-publish using a different randomly picked routing key</li>

<li>Have your consumers connect to all queues</li></ul></div><div>Using the technique described above, you can load balance incoming TCP connections to a cluster of rabbit nodes. Load will get distributed based on the randomly picked routing keys used by the publishers.</div>
