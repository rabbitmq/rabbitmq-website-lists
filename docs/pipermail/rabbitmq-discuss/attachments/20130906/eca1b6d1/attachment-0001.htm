So I assume the problem is that you don&#39;t think these connections should be listed, since you think they were short-lived not-really-AMQP connections from a monitoring tool?<br>
<br>
Connections which have been closed (abruptly or not) should really not make it to this list. You might want to attempt to correlate the connection pid (e.g. &quot;&lt;0.470.0&gt;&quot; from the log above) with the results of &quot;rabbitmqctl list_connections user peer_host peer_port pid state&quot;. If you see the same pid in both, that&#39;s a bug.<br>

<br>
But actually I think those are real client connections that have somehow been routed to your host B, since the ones from zenoss should never have got as far as the &#39;running&#39; state.<br>
<br>
&quot;rabbitmqctl list_connections user peer_host peer_port client_properties&quot; might be illuminating - this will add a column listing the self-declared client properties to the output, only real AMQP clients will have such properties.<br>
