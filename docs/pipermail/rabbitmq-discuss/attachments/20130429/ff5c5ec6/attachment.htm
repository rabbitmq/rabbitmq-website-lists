<html><head><meta http-equiv="Content-Type" content="text/html charset=us-ascii"></head><body style="word-wrap: break-word; -webkit-nbsp-mode: space; -webkit-line-break: after-white-space; ">Ryan,<div>First of all, apologies for not responding more quickly.</div><div><br></div><div>You may have noticed that some support for a JMS client using RabbitMQ as the broker is now available from vFabric Suite Release 5.3 [1], as part of vFabric RabbitMQ 3.0[2].</div><div><br></div><div>What is offered in this first version (RabbitMQ JMS 1.0.0) is a Java Client and an extra RabbitMQ plugin that together offer the core JMS functions at the level of JMS Spec 1.1.</div><div><br></div><div>The key features&nbsp;<i>not </i>offered&nbsp;are:</div><div>&nbsp; &nbsp; Full transaction support (local transactions <i>are</i> supported without full recovery)</div><div>&nbsp; &nbsp; Queue Browsing</div><div>&nbsp; &nbsp; Queue Selection</div><div>&nbsp; &nbsp; Exception Listeners</div><div>&nbsp; &nbsp; Connection Consumers and Server Sessions.</div><div><br></div><div>The JMS Admin interface can be used to define Destinations and create Connections and these are serialisable and can be delivered over JNDI to app code.</div><div><br></div><div>The JMC Client interface can send messages to and receive messages from topics and queues using the JMS interface (including the use of topic selection). The messages generated are structured specifically for JMS client use, in particular they use Java serialisation for payloads. &nbsp;This means that there is little help for a standard RabbitMQ client to read the messages without some extra 'decoding' work, though for raw binary payloads it would not be difficult. &nbsp;The queue names and exchanges are JMS client specific, but not difficult to discover.</div><div><br></div><div>It is quite a different story for a non-JMS client to send messages <i>to</i> a JMS Destination. &nbsp;This would involve quite a lot of 'coding', as there is nothing in the JMS client to allow the definition of RabbitMQ Queues as Destinations for reading.</div><div><br></div><div>While there are plans to subsequently extend this support to include some of the missing items above, there is nothing in plan yet for smooth interoperation between JMS and non-JMS RabbitMQ clients. &nbsp;We are presently evaluating demand.</div><div><br></div><div>Currently, the first service release is being developed to fix outstanding bugs and this will be made available at some point in the (near) future.</div><div><br></div><div>Your input is very valuable to us. &nbsp;If you have specific requirements please make them known, either on this forum as usual, or to me directly on &lt;spowell at <a href="http://gopivotal.com">gopivotal.com</a>&gt;.</div><div><br></div><div>[1]&nbsp;<a href="https://www.vmware.com/support/vfabric-platform/doc/vfabric-suite-rn-5.3.0.html">https://www.vmware.com/support/vfabric-platform/doc/vfabric-suite-rn-5.3.0.html</a></div><div>[2]&nbsp;<a href="https://www.vmware.com/support/vfabric-rabbitmq/doc/vfabric-rabbitmq-rn-3.0.4.html">https://www.vmware.com/support/vfabric-rabbitmq/doc/vfabric-rabbitmq-rn-3.0.4.html</a></div><div><br></div><div><div apple-content-edited="true">
